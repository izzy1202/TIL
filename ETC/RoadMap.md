## 1. 인터넷과 HTTP
- 네트워크 : '네트' + '워크'
  - 여러 대의 컴퓨터가 서로 통신할 수 있게 해주는 것
- 인터넷 : 네트워크들의 집합체(네트워크의 네트워크)

### 인터넷 작동 원리
[강의 소개 링크 - 인터넷은 어떻게 동작하는가?](https://developer.mozilla.org/ko/docs/Learn/Common_questions/Web_mechanics/How_does_the_Internet_work)
  - 여러 대의 컴퓨터를 연결하기 위해 '라우터'라는 것을 사용한다.
  - 라우터끼리도 연결함으로써 네트워크를 확장시킬 수 있다.
  - 네트워크를 전화 시설과 연결하기 위해 모뎀이 필요하다.
  - ISP(Internet Service Provider) : 인터넷을 제공하는 업체
  - 네트워크에 연결된 모든 컴퓨터에는 IP 주소 (IP는 (Internet Protocol)라는 고유한 주소가 있다. 주소는 점으로 구분 된 네 개의 숫자로 구성된 주소이다 . 예: 192.168.2.10.
    - DNS(도메인) : 사람이 알아보기 쉽게 알파벳이나 한글 등으로 변경한 것
    
### 웹소켓 
- 웹소켓은 데이터를 HTTP로 주고 받는데, HTTP는 클라이언트가 서버에 요청을 해야지만 응답을 줄 수 있다.
    - 웹소켓은 이런 단점을 보강하기 위한 프로토콜로써 HTTP위에서 생성되었다.

## 2. 버전 컨트롤
- Git
  - 분산형 버전 관리 시스템
  - 3가지 영역
    1. 작업 디렉토리(워킹 디렉토리)
    2. 스테이지 에어리어 : commit 하기위해 git add명령어로 추가하면 올라가는 영역
    3. 레포지토리 : git commit하면 올라가는 영역
      - commit 후 push까지 하면 원격 서버에 commit 파일들을 업로드하게 된다.
- GitHub
  - Git 기반의 호스팅 사이트
  - Git에서 git push로 파일을 올린 원격 서버가 Github
  - Pull requests : 관리자가 아닌 사람이 코드의 변경 요청을 할 때 사용하는 메뉴
  - Actions : 코드 저장소에 어떤 이벤트가 일어날 때 특정 작업이 일어나게 하거나, 주기적으로 어떤 작업들을 반복시켜서 실행하고 싶은 경우에 사용한다.
  - Projects : 프로젝트 관리하는 메뉴
  - Wiki : 해당 프로젝트의 위키 문서들을 저장할 수 있는 공간
  - insights : 이 저장소가 어떤 식으로 운영 되고 있는지 통계 수치를 보여준다.

 ## 5. 리눅스 명령어
- pwd(print working directory) : 현재 작업 위치를 출력하는 명령어
- ls(list Segment) : 현재 디렉토리의 파일과 디렉토리를 보여준다.
- cd(change directory) : 디렉토리 이동
- mkdir : make directory
- cp : copy
- mv : move directory
- rm : remove directory
- cat : 파일의 내용 확인
- touch : 빈 파일 생성
- echo : 화면에 글자 표시
- ip addr / ifconfig: 아이피 정보 확인
- ss : 네트워크 상태 확인
- nc(net cat) : 서버의 포트 확인
- which : 어떤 명령어가 어디에 있는지 찾을 때 사용
- tail : 파일의 마지막 부분을 보여주는 명령어
- find : 파일이나 디렉토리 찾는 명령어
- ps : 프로세스 상태를 보여준다
  - ps -aux
- grep : 어떤 파일에서 패턴을 찾아서 파일의 내부에 패턴을 만족하는 문자열이 있는지 확인한다.
- kill : 프로세스를 죽이는 명령어
  - ps -aux해서 두번째줄의 프로세스 아이디로 죽일수 있다.

## 8. 프레임워크
- 프레임워크 : 개발시에 필요한 것들을 미리 만들어 두어서 작업을 편하게 해주는 코드 덩어리이다.
- 2가지 종류
  1. 개발시 설정을 어떻게 하고 코드를 어떻게 짜야 하는지 제안하는 프레임워크
  2. 최소한의 코어만을 가지고 설정과 제약을 조금만 하는 프레임워크
- 3가지 소개
  - 모두 서버측에서 API를 만드는 데 필요한 프레임워크들이다.
  1) NestJS
    - Node js에서는 익스프레스가 대표적 프레임워크로, 익스프레스는 최소한의 코어를 가지고 확장해나가는 프레임워크이다.
    - 반면, NestJS는 아키텍처와 구조를 잡아주는 프레임워크이다.
    - 데코레이터를 사용하여 추가적인 설정 파일을 두지 않고 라우터나 미들웨어를 설정할 수 있다.
    - 내부적으로는 익스프레스와 패스트파이를 사용하고 있기 때문에, 기존의 익스프레스의 미들웨어를 사용할 수 있는 장점이 있다.
    - 아직까지는 자바의 스프링에 비해 기업용 프로그램을 작성하기엔 부족한 부분이 있지만, 최근 익스프레스의 발전이 더디기 때문에 강사님 개인적으로 NestJS가 조금 더 발전 가능성이 있다고 생각하여 추천한다.
  2) spring
    - Java
    - 기업용 애플리케이션을 만드는 데 필요한 것들을 모두 갖추고 있는 프레임워크이다.
    - 스프링 웹플럭스라고 불리는 리액티브 프로그래밍에서는 로그 추적이 어렵고 멀티코어 활용이 어렵다는 아쉬운 부분이 존재하긴 하지만, 기업용 애플리케이션 작성에 최적화된 프레임워크이다.
    - 자바가 발전이 느린 편이며 [행사 코드](https://seungdols.tistory.com/538)가 많지만, 최근 코틀린이 나와서 어느정도 해소되었다.
  3) FastAPI
    - Python 커뮤니티에서 최근 인기있는 프레임워크이다.
    - Python에는 장고와 플라스크라는 유명한 프레임워크가 있지만, API만 작성하는 용도라면 FastAPI를 추천한다.
    - 최소한의 설정과 제약을 주는 프레임워크로, 플라스크와 비슷한 컨셉의 프레임워크이다.
    - 스프링과 비슷하게 의존성 주입을 사용한다.
    - 파이덴틱 기반의 유효성 검증이 좋다.
    - 스웨거로 되어 있는 문서 자동화도 잘 되어 있다.
    - 어싱크 어웨이크 기반의 동시성을 도입해서 성능이 좋고 배우기 쉽다. 

- 백엔드 개발을 하기 위해서는 하나는 깊게 공부해야 한다.

## 9. 데이터베이스
- 데이터베이스 : 검색과 축적이 쉽도록 정리된 데이터의 모음으로, 데이터베이스의 소프트웨어를 의미한다.
- 데이터 저장 시 수많은 문제가 생길 수 있는 상황에서 최대한 안전하게 데이터를 보관/검색/수정/삭제할 수 있게 해주는 소프트웨어이다.
- RDB와 RDB가 아닌 것으로 구분된다.
  1. RDB(Relational DataBase)
    - 데이터를 행과 열로 이루어진 테이블로 관리한다.
    - 기본키를 사용해서 각 행을 식별한다.
    - 각 테이블간의 관계를 지울 수 있다.
    - ex) Oracle, MySQL, SQL서버, PostgreSQL 등
  2. NoSQL
    - No Sql이라는 의미로 쓰이기도 하지만, Not Only Sql(SQL뿐만 아니라 다른 것을 더 쓴다)로 더 많이 쓰인다.
    - RDB는 데이터의 저장시 수정/삭제가 용이하지만, 성능을 높이려면 분산 서버를 생각해야 한다. NoSQL은 RDB에 비해 이런 부분이 잘 되어있다.
        - MongoDB : 도큐먼트를 저장하는 DB
          - 데이터로 JSON과 매우 비슷하게 생긴 BSON이라는 문서 모델을 저장한다.
          - 클러스터를 지원해서 확장이 용이하고, 많은 요청이 오는 경우에도 속도가 늦어지지 않도록 지원한다.
        - redis : 키 밸류 기반의 NoSQL 데이터베이스이다.
          - 보통 분산 캐시로 많이 사용하는데, 키를 기반으로 데이터를 잘 관리할 수 있도록 해주기 때문에 데이터베이스의 범주에 넣었다.
          - list, set, hash 등의 다양한 데이터 타입을 지원하고 있고 빠른 성능을 내주기 때문에 대부분의 회사에서 기본적으로 사용하고 있다.
- DB 지식들
  - ORM(Object Relation Mapping) : 클래스 오브젝트와 테이블의 관계를 클래스로 가져올 수 있도록 해주는 프로그램
    - 특정 클래스의 메소드를 실행하면 실행시에 쿼리 생성을 하거나 쿼리를 실행해준다.
    - 쿼리 작성 없이 DB 조작이 가능하며, 직관적으로 데이터를 관리할 수 있다.
    - 통계 쿼리같은 복잡한 쿼리는 RAW 쿼리를 사용하는 것이 낫다.
  - ACID
    - 원자성, 일관성, 격리성, 내구성을 의미
    1) 원자성 : 트랜잭션을 구성하는 명령이 하나의 묶음으로 처리되어서 모두 성공하거나 모두 실패하는 것을 보장하는 기법이다.
    2) 일관성 : 트랜잭션에서 사용된 변경사항이 데이터베이스의 무결성조건을 만족할 수 있도록 하는 것이다.
    3) 격리성 : 두 개의 트랜잭션이 서로에게 영향을 미칠 수 없다는 것을 의미한다.
    4) 내구성 : 트랜잭션이 성공적으로 끝나면 그 결과가 데이터베이스에 계속 유지된다는 것을 의미한다.
    - 데이터베이스에서 데이터의 일관성을 유지할 수 있도록 해주며, 문제 발생시에 이전으로 돌릴 수 있도록 해준다.
   - 트랜잭션 : 데이터베이스의 상태 변화를 시키기 위해서 수행하는 작업의 단위이다.
      - 트랜잭션을 씀으로써 동시다발적으로 발생하는 데이터 변경 요청중에도 DB의 데이터가 안전하게 변경될 수 있도록 해준다.
   - N + 1 
    - ORM 사용 시 많이 마주하는 문제이다.
    - 클래스의 모델에 다른 클래스의 관계를 설정하고 해당 클래스를 조회할 때, 데이터 개수만큼 쿼리가 발생하는 것을 말한다.
    - 조인을 사용하여 한 번에 데이터를 가져오는 것을 의도했으나 n + 1번 실행돼서 불필요하게 많은 쿼리가 발생하는 것을 의미한다.
  
## 10. API에 대하여
  - 2가지 소개
  - 1) REST
      - Representational State Transfer(표현적 상태 전달자)
        - 자원의 표현에 의한 상태 전달
      - 자원은 HTTP URI 방식으로 자원을 표현한다.
      - REST는 HTTP Method를 사용하여 해당 자원에 대한 상태를 전달해주는 것이다.
      - POST(데이터 추가), GET(데이터 가져옴), PUT(업데이트), DELETE(삭제)
      - API는 REST 방식으로 작성한다.
      
      ![image](https://user-images.githubusercontent.com/106478906/231480692-223c0899-33f4-4fe3-975f-526eecb5dadb.png)

       - 자원은 URL의 HTTP URI 방식으로 표시하고, 그 자원에 대한 상태를 가져오고, 변경하고 싶을 때 HTTP 메소드(POST, GET, PUT, DELETE)를 쓴다.
 - 2) GraphQL
    - 쿼리언어 스펙
    - JSON 형태
    - 클라이언트가 서버로부터 데이터를 효과적으로 가져오게 하는 것이 목적이다.
    - REST는 서버가 쿼리를 DB에 날려서 데이터를 가져오고,
  GraphQL은 클라이언트가 서버에 쿼리를 날린다.
    - endpoint가 하나만 있다.
    <img width="461" alt="image" src="https://user-images.githubusercontent.com/106478906/231481801-488b2c1c-2430-4ab4-9c6b-84f42e032786.png">
    
      -> REST같은 경우는 각각의 URL을 계속 만들어줘야 하는데, GraphQL은 endpoint가 /graphql 딱 하나이다.
      -> 쿼리를 어떻게 짜냐에 따라서 어떤 데이터를 가져올지 결정할 수 있다.

## 12. 배포하기
- 보통 배포 도구를 사용한다.
- 배포 도구가 없다면? FTP를 이용해서 압축 파일을 수동으로 서버에 업로드한다. 하지만 실수하기 쉽기 때문에 도구를 사용하는 것이 좋다.
- 단순하고 반복적인 작업이지만, 절대로 실수하면 안되는 매우 중요한 작업이다.
- 배포의 2가지 원칙
1. CI : 지속적 통합
  - 배포 전의 과정을 자동화하는 것이다.
  1) 코드 병합
  2) 코드 빌드
  3) 코드 테스트
2. CD : 지속적 배포
  - 코드 작성 이후에 코드가 서버로 나갈 때까지의 활동들을 자동화하는 것이다.
  1) 스테이징 배포 
  2) 수동 배포
  3) 자동배포
  - 롤링 배포 : 서버 여러대 중에 몇 대만 계속 돌아가면서 배포하는 것이다.
  - 블루/그린 배포 : 똑같은 수의 서버를 미리 다 띄워두고 주소만 바꿔치기 한다.
  - 까나리 배포 : 특정 퍼센트의 서버에만 배포한 다음 문제가 없으면 점차 배포를 늘려간다.
